### 1. Runtime Ownership Boundary
- **Recommended:** Extract a dedicated `doc-web` repository that Dossier consumes as a versioned library. This cleanly separates the semantic-HTML building layer from both codex-forge (the R&D scanner) and the Dossier application. A standalone repo lets the doc-web team maintain its own CI/tests, issue triage, and versioning without dragging along codex-forge’s research baggage. It enforces a clear “black box” contract: codex-forge emits HTML+manifest, doc-web assembles it, and Dossier uses the result. This model maximizes parallel work (codex-forge folks iterate on extraction, doc-web folks on site-building, Dossier folks on integration) and makes ownership explicit.
- **Runner-up:** Integrate the doc-web code directly into Dossier. This reduces overhead if doc-web is only ever used by Dossier and never shared. In that case the product team can develop the web-builder alongside Dossier features. This may be simpler in the short term (one repo, one pipeline), but it entangles responsibilities (presentation-layer code lives inside the Dossier codebase) and makes sharing or reuse harder. It’s only advisable if the doc-to-site logic is trivial or Dossier is the sole consumer.
- **Avoid:** Leaving doc-web inside codex-forge (or a lightly pruned fork) as a permanent runtime. Codex-forge is fundamentally an R&D prototype/pipeline (with gamebook-specific logic) and is not designed for stable, production-style releases. Elevating it to a versioned library risks carrying along irrelevant complexity and unstable APIs. In practice, mixing prototyping code with production dependencies often leads to hidden coupling, brittle interfaces, and accumulated technical debt. As noted in distribution strategies, separating distinct components into their own repos yields stronger ownership and organization ([www.petermcconnell.com](https://www.petermcconnell.com/posts/monolith_vs_microservice_repos/#:~:text=Strong%20Ownership%3A%20with%20microservice%20repositories,too%20many%20external%20teams%20beforehand)) ([www.petermcconnell.com](https://www.petermcconnell.com/posts/monolith_vs_microservice_repos/#:~:text=Better%20Organization%3A%20With%20each%20service,the%20system%20on%20their%20own)). Embedding doc-web in codex-forge would violate this by merging two disparate concerns.
- **Evidence:** Architecturally, multi-repo (“microservices”) setups favor clear module boundaries and owner independence. For example, teams often break functionality into separate repos so each can use independent tooling, CI, and versioning ([www.petermcconnell.com](https://www.petermcconnell.com/posts/monolith_vs_microservice_repos/#:~:text=Strong%20Ownership%3A%20with%20microservice%20repositories,too%20many%20external%20teams%20beforehand)) ([www.petermcconnell.com](https://www.petermcconnell.com/posts/monolith_vs_microservice_repos/#:~:text=Better%20Organization%3A%20With%20each%20service,the%20system%20on%20their%20own)). In Peter McConnell’s analysis, separate repos give “strong ownership” and “better organization” to each component ([www.petermcconnell.com](https://www.petermcconnell.com/posts/monolith_vs_microservice_repos/#:~:text=Strong%20Ownership%3A%20with%20microservice%20repositories,too%20many%20external%20teams%20beforehand)) ([www.petermcconnell.com](https://www.petermcconnell.com/posts/monolith_vs_microservice_repos/#:~:text=Better%20Organization%3A%20With%20each%20service,the%20system%20on%20their%20own)). By analogy, making `doc-web` its own repo follows this pattern. Conversely, a monolithic repo simplifies some tooling but “requires more cognitive load” and can bloat when all teams share it ([www.petermcconnell.com](https://www.petermcconnell.com/posts/monolith_vs_microservice_repos/#:~:text=Better%20Organization%3A%20With%20each%20service,the%20system%20on%20their%20own)). Thus a standalone `doc-web` repo (with tagged releases) best balances maintainability, clarity, and parallel development.

### 2. Output Contract for Structural Website
- **Recommended:** The `doc-web` output should be a well-structured static website bundle. At minimum it should contain: a root `index.html` (table-of-contents or landing page), one HTML file per chapter/section/page, and any image assets in subfolders. Each HTML page should be semantically marked up (using headings, paragraphs, lists, etc.) but carry no heavy styling or theme. Include minimal navigation links (e.g. “Next”/“Previous” and a TOC link) to aid human browsing. Crucially, include a **manifest file** (e.g. JSON or YAML) that enumerates all pages, their titles, source-page mappings, and provenance fields. This manifest (or sidecar JSON) lets downstream tools programmatically understand the site structure and back-link content to sources. Finally, emit **provenance sidecars** (JSON) mapping HTML elements to original source pages/coords. This bundle – plain HTML + manifest + assets – ensures both AI (text parsers) and humans (browsers) can consume it effectively.
- **Runner-up:** A more “bare” site: only HTML pages and images, with no explicit manifest or navigation. This would rely on implicit ordering (e.g. filenames) for hierarchy. It’s technically minimal and still usable (a scraper could infer sequence), but it places more burden on the consumer to guess structure. Without a manifest, provenance data would have to live in HTML `data-` attributes only. This might be acceptable for the simplest cases, but risks ambiguity.
- **Avoid:** Embedding full presentation/theming. The doc-web contract should **not** prescribe site styling (CSS, fonts, etc.) beyond bare-bones layout. Do not bake in a “pretty” template. The idea is structural output only. Adding complex templates or front-end frameworks would blur responsibilities with Dossier’s theming layer. Likewise avoid proprietary or binary artifacts; stick to open formats (HTML, JSON).
- **Evidence:** Static site generators operate on similar principles. For example, Hugo (a popular SSG) builds a final `public/` folder containing `index.html`, error pages, and nested directories of HTML files for each section ([chrisdaaz.github.io](https://chrisdaaz.github.io/static-web-scholcomm/tutorials/static-site-generators/#:~:text=,html)). In one sample, Hugo’s `public/` contains `404.html` at root and a `categories/index.html` subpage ([chrisdaaz.github.io](https://chrisdaaz.github.io/static-web-scholcomm/tutorials/static-site-generators/#:~:text=,html)), illustrating a simple yet complete static output layout. By analogy, our site bundle should include top-level pages and subdirectories as needed. Including a manifest is akin to an EPUB or web-app manifest – it provides machine-readable structure and provenance. The Hugo example confirms that a robust static bundle includes an entry point plus hierarchically organized content ([chrisdaaz.github.io](https://chrisdaaz.github.io/static-web-scholcomm/tutorials/static-site-generators/#:~:text=,html)), backing up the recommended structure.

### 3. Provenance and Citation Contract
- **Granularity:** Default to *block-level* provenance (e.g. each paragraph, table, or figure caption). Each HTML element that could carry a fact or quote should be mappable to the source. Finer granularity (e.g. sentence-level) is ideal but may not always be practical; paragraphs are a good default unit. This enables attaching citations to specific blocks.
- **Stable Identifiers:** Assign each prose block a stable `id` or `data-*` attribute for referencing. For example, `<p id="block-ch1-5" data-provenance-page="42">`. The `id` should be reproducible (e.g. based on chapter/page and offset) so that it doesn’t change on rebuild. Use either DOM `id` or a custom `data-provenance-id` to tag each block. This ID will be used in sidecar mappings.
- **Sidecar Format:** Use JSON (or JSON-LD) to map HTML IDs to original locations. A robust approach is the W3C Web Annotation style (targeting segments with URL fragments). For instance, an entry might look like:

  ```json
  {
    "block-ch1-5": {
      "source": "book.pdf",
      "page": 42,
      "coords": [100,200,300,50]
    }
  }
  ```

  Here `coords` is optional (a bounding box on the PDF page). One could formally use `pdfurl#xywh=x,y,w,h` fragments – e.g. an image annotation ID like `http://example.com/page#xywh=100,100,300,300` ([www.w3.org](https://www.w3.org/TR/2016/CR-annotation-model-20160705/#:~:text=%22id%22%3A%20%22http%3A%2F%2Fexample.com%2Fimage1,)) is a common pattern. Similarly, storing `page=42` plus `#xywh` in JSON or as a fragment aligns with these best practices.
- **Mandatory vs. Optional Fields:** At minimum, each record must include the stable HTML ID, the source document ID (e.g. filename “book.pdf”), and the source page number. These are mandatory for citation. Optionally include coordinates or text-offsets if available; this allows opening the PDF at the exact location. The source element type (e.g. “Paragraph” or “Figure”) could also be included but is secondary. In summary, require `{id, source, page}`, with `{coords, type}` optional.
- **Evidence:** The W3C Annotation Data Model illustrates using URL fragments for exact targeting. For example, an annotation ID like `http://example.com/image1#xywh=100,100,300,300` pinpoints a rectangle on an image ([www.w3.org](https://www.w3.org/TR/2016/CR-annotation-model-20160705/#:~:text=%22id%22%3A%20%22http%3A%2F%2Fexample.com%2Fimage1,)). By extension, one can use `source.pdf#xywh=` to denote regions in a PDF. This shows the viability of storing coordinates. Ensuring each HTML element has a matching ID allows sidecars to reference it. The use of `#xywh` for coordinates is a recognized practice ([www.w3.org](https://www.w3.org/TR/2016/CR-annotation-model-20160705/#:~:text=%22id%22%3A%20%22http%3A%2F%2Fexample.com%2Fimage1,)). Thus a JSON sidecar mapping HTML IDs to `page` and `#xywh` (or explicit coords) follows established patterns for granular provenance.

### 4. Packaging and Versioning Model for Dossier
- **Recommended:** Publish `doc-web` as a versioned package with clear releases. Use git tags and semantic versioning (e.g. v1.0.0, v1.1.0). Dossier should depend on specific version pins (via pip or a Git tag) rather than a moving branch. Maintain a CHANGELOG that notes API contracts and breaking changes. Ideally, set up CI to run doc-web’s tests and produce artifacts on each release. If an internal package index or PyPI is available, host `doc-web` there; otherwise rely on git. This ensures Dossier can upgrade controllably.
- **Runner-up:** Vendor or “git submodule” the doc-web code inside Dossier. That means Dossier routinely pulls in updates when convenient. This avoids full packaging overhead, but it effectively copies code. Only do this if you cannot maintain a shared repo. Even here, you should submodule a tagged release so changes are explicit.
- **Avoid:** Pointing Dossier at an unstable branch or “latest” from development. Without version discipline, Dossier may break when doc-web’s interface changes underfoot. Also avoid turning doc-web into an internal library without version records; lack of tagging = no traceability. Similarly, avoid double-publication models like having one team push and another to sync manually.
- **Contract Tests:** Build integration tests for the doc-web ↔ Dossier boundary. For example, Dossier can have a test that runs doc-web on a sample HTML input and asserts certain output structure (e.g. presence of `index.html`, or that provenance JSON has expected fields). These “consumer-driven contract” tests catch interface drift. The fixture could be a small PDF or HTML excerpt with known output. Automated tests (on doc-web’s side as well) ensure compatibility.
- **Release Discipline:** Adopt Semantic Versioning rules (major bumps for breaking changes) so that Dossier can lock to a major version if needed. Following established release processes prevents silent breakage. (For instance, OpenLMIS components are “each semantically versioned” and posted as tagged releases ([openlmis.readthedocs.io](https://openlmis.readthedocs.io/en/rel-3.4.0/conventions/versioningReleasing.html#:~:text=The%20components%20are%20each%20semantically,service%20components%20and%20UI%20components)) ([openlmis.readthedocs.io](https://openlmis.readthedocs.io/en/rel-3.4.0/conventions/versioningReleasing.html#:~:text=All%20OpenLMIS%20source%20code%20is,on%20Jenkins%20forever%20by%20default)).) Mandate that any change affecting the doc-web output schema requires at least a minor/major version increment and notes in the changelog. This transparent versioning and tagging practice forestalls hidden incompatibilities.
- **Evidence:** Large componentized systems illustrate this model. For example, OpenLMIS (a microservices-based platform) maintains separate repos for each component, uses semantic versioning, and tags every release ([openlmis.readthedocs.io](https://openlmis.readthedocs.io/en/rel-3.4.0/conventions/versioningReleasing.html#:~:text=The%20components%20are%20each%20semantically,service%20components%20and%20UI%20components)) ([openlmis.readthedocs.io](https://openlmis.readthedocs.io/en/rel-3.4.0/conventions/versioningReleasing.html#:~:text=All%20OpenLMIS%20source%20code%20is,on%20Jenkins%20forever%20by%20default)). Releases of those components (and of their reference distribution) are explicitly tagged on GitHub ([openlmis.readthedocs.io](https://openlmis.readthedocs.io/en/rel-3.4.0/conventions/versioningReleasing.html#:~:text=All%20OpenLMIS%20source%20code%20is,on%20Jenkins%20forever%20by%20default)). By analogy, `doc-web` should be treated similarly: its own repo with tagged releases (perhaps published to an artifact registry), so Dossier can safely pull known-good versions. SemVer-backed releases and changelogs are proven practices for avoiding “silent” breakages ([openlmis.readthedocs.io](https://openlmis.readthedocs.io/en/rel-3.4.0/conventions/versioningReleasing.html#:~:text=The%20components%20are%20each%20semantically,service%20components%20and%20UI%20components)) ([openlmis.readthedocs.io](https://openlmis.readthedocs.io/en/rel-3.4.0/conventions/versioningReleasing.html#:~:text=All%20OpenLMIS%20source%20code%20is,on%20Jenkins%20forever%20by%20default)).

### 5. Migration Strategy from codex-forge
- **What to move “as is”:** Take over the parts of codex-forge that format and assemble HTML and site structure. For instance, the code in `modules/build/build_chapter_html_v1/main.py` is the core HTML builder – it should be copied (or refactored) into doc-web. Likewise, any manifest-writing logic and simple templating (header/footer insertion) can be migrated. Essentially, move the logic that inputs semantic fragments and outputs static pages.
- **What to refactor first:** Remove any codex-specific hooks or flags. For example, strip out any “Fighting Fantasy” gamebook logic or special metadata handling that isn’t generic. Refactor so that the code takes a neutral “document” input rather than an FF dataset. Also externalize configuration: codex-forge may have hardcoded paths or conventions (e.g. table of contents markers); make these parameters. Ensure there are no remaining dependencies on codex modules (OCR, ML models, etc.) – replace them with stubs or interfaces. Essentially generalize the code to “any book,” not just specific collections.
- **What to leave behind:** Heavy research logic and domain-specific heuristics should **not** come along. For example, FF puzzle-solving logic, gamebook flow, or campaign randomizer code are out. Also, image processing, OCR, and AI-based text extraction should remain in codex-forge. These concern how content is produced, not how it’s served. Doc-web’s job is presentation, so skip anything not essential to layout or linking.
- **Fixture/golden handling:** Before archiving codex-forge, migrate any golden test fixtures that pertain to the build output. Doc-web should inherit a minimal set of example inputs (e.g. a small PDF and its expected HTML) to serve as regression tests. If codex-forge is archived, ensure these fixtures live either in doc-web or in Dossier’s tests. In practice, copy the relevant HTML+manifest pairs into the new repo so they can be used for CI.
- **Small first extraction slice:** As a proof-of-concept, start by extracting only the simplest functionality: generate an `index.html` linking chapter pages, and produce those pages with minimal content. For example, port the part of `build_chapter_html_v1` that iterates chapters and writes files, without worrying about styling or advanced links. This limited slice will expose dependencies and allow refactoring. Once the minimal end-to-end flow works (e.g. “running doc-web on sample data produces a browsable site with Manifests”), gradually add back navigation links, manifest fields, and provenance tags. This incremental lift minimizes risk and proves the boundary.
- **Evidence:** (General software engineering practice.) The process described follows standard refactoring/“extract library” guidelines. For instance, techniques like “Strangler Fig” or component extraction suggest first migrating only stable, well-understood code, then refactoring intertwined logic. By carving out build/output code from codex-forge and keeping it lean, we preserve the codex-forge codebase for experimentation while giving doc-web a clean, maintainable core. This approach is routinely recommended when moving features from a prototype repo into a production service.
