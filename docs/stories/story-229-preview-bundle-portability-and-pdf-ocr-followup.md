# Story 229 — Harden Preview Bundle Portability and Mixed-PDF OCR Fallback

**Priority**: High
**Status**: Done
**Decision Refs**: `docs/decisions/adr-002-doc-web-runtime-boundary/adr.md`, `docs/doc-web-bundle-contract.md`, `docs/dossier-doc-web-handoff.md`
**Depends On**: Story 228

## Goal

Close the material preview-contract defects found by Storybook after Story 228. The preview bundle must not silently under-read image-only sampled PDF pages after a text-layer first page, and portable preview snapshots must be persistable/replayable from manifest-declared safe files and privacy-safe cache identity rather than local source paths, temp paths, storage keys, or donor filenames.

Storybook's report decomposes into three material follow-up issues:

- mixed text-layer plus image-only PDFs could silently skip sampled image-only pages after page 1;
- portable preview snapshots did not explicitly declare the bundle-local files safe to persist and replay;
- replay/cache identity and persisted safe files could still depend on local paths, storage keys, source filenames, donor filenames, display labels, or OCR-error/source-element details derived from private source names.

## Eval Ladder Context

- **Eval ladder**: This is a deterministic contract/runtime follow-up, not a new AI capability eval. The relevant parent proof is Story 228's preview bundle mode with schema validation and real sample outputs. The measured failure mode is contract/orchestration risk: page 2+ OCR fallback, safe-file declaration, and privacy-safe cache identity. Focused pytest + schema validation + manual artifact inspection are the correct validation surface; no promptfoo eval is needed unless future work changes OCR/model quality.

## Acceptance Criteria

- [x] A checked-in fixture PDF has page 1 with usable text-layer content and page 2 as image-only content.
- [x] `doc-web preview` extracts page 2 text through OCR fallback and emits page/block provenance pointing to source page 2.
- [x] Missing, failed, or timed-out `pdftoppm -singlefile` raster output is a hard visible failure, not a silent no-text page or private-path leak.
- [x] Scan-heavy sampled PDF OCR fallback respects the global preview budget and reports `preview_ocr_budget_exhausted` as partial/deferred coverage instead of causing load-sensitive hard preview timeouts.
- [x] Empty OCR fallback on a sampled PDF page produces unique skipped-page metadata with the sampled-page failure reason preserved, not duplicate page rows.
- [x] `manifest.json` declares which bundle files are safe to persist/replay, uses bundle-relative file and asset-root paths, and excludes or marks non-portable/private/debug/cache-local files.
- [x] Manifest schema validation rejects unsafe source refs, non-portable file/asset-root paths, missing replay-required rows, unsafe replay claims, and debug/private/cache-local rows marked as portable or safe.
- [x] Manifest schema validation rejects required file path/role mismatches so core replay paths cannot be mislabeled as generic assets.
- [x] Portable safe files do not contain absolute local paths, temp paths, source document paths, storage keys, or source filenames required for replay.
- [x] Timeout/failure CLI JSON emits bundle-relative failure artifact references and does not leak the local output directory.
- [x] CLI failed status rows preserve the specific failure reason (`timeout` or `pdf_rasterization_*`) instead of reporting every preview failure as a generic timeout.
- [x] The preview runtime contract remains available through both module entry surfaces: `python -m doc_web` and `python -m doc_web.cli`.
- [x] Cache identity includes stable privacy-safe fields: doc-web version/ref, parser/OCR settings, page count, bundle fingerprint, and source hash/ref.
- [x] Cache identity is a first-class validated contract surface: `doc_web_cache_identity_v1` is accepted by `validate_artifact.py`, included in preview metadata schema validation, and exposed through the runtime contract schema versions/fingerprint.
- [x] Cache-identity schema rejects colon-form storage/source references such as `s3:...`, `azure:...`, and `file:...`, rejects partial or malformed `sha256:` fingerprint/cache-key strings, rejects stale `identity_fingerprint` values, rejects embedded donor filenames, restricts display-label path exemptions to `source_identity.source_display_label`, normalizes integer/float content-hint timeout inputs to the same identity, and still allows identity values that are exactly `sha256:<64 lowercase hex>` plus valid colon-form content-hint model IDs.
- [x] Default `content_hint_mode=auto` and AI content-hint fallback errors cannot make cache identity validation fail or leak raw private fallback details into manifest-declared safe metadata.
- [x] Identical source/runtime/parser runs produce the same portable identity, while changed parser/runtime settings change it.
- [x] Display/source labels can be rewritten by consumers without invalidating structural provenance because identity no longer depends on local filenames.
- [x] Raw PDF/DOCX metadata titles or creators that look like local paths, storage refs, donor filenames, or punctuated donor filename references are sanitized before entering manifest-declared portable safe files.
- [x] Raw PDF/DOCX metadata titles or creators that match the source filename stem are sanitized before entering manifest-declared portable safe files.
- [x] DOCX title paragraphs that match the source filename stem are sanitized before entering manifest-declared portable safe files or default entry titles.
- [x] Caller-provided `run_id` values are validated as portable identifiers and rejected when path-like or embedded source-filename-shaped before they enter manifest-declared portable safe files.
- [x] Image-directory safe files/source element ids and PDF/image OCR-error metadata use portable page/error references rather than source image filenames or paths.

## Out of Scope

- Broad OCR quality improvement beyond bounded preview fallback.
- Changing final maintained PDF recipe coverage or claiming a new final format-support row.
- Dossier or Storybook implementation changes.
- Persisting raw source bytes or private debug/cache-local files inside the portable snapshot.
- Replacing the accepted `doc-web` bundle/provenance boundary from ADR-002.

## Approach Evaluation

- **Simplification baseline**: A single LLM call cannot own Poppler raster filename semantics, manifest safe-file classification, or cache-key privacy. This is runtime contract and artifact plumbing.
- **AI-only**: Not appropriate. AI could summarize preview text, but it cannot guarantee deterministic page rasterization, replay identity, or path privacy.
- **Hybrid**: Not needed for this slice. Detection is deterministic: text-layer extraction per sampled PDF page, OCR fallback only for textless sampled pages, and schema-backed portable files.
- **Pure code**: Selected and implemented. The problem is orchestration, metadata, and reproducible artifact contracts.
- **Repo constraints / prior decisions**: ADR-002 keeps `doc-web` as the standalone structural bundle/provenance boundary for Dossier and Storybook. Story 228 made preview mode part of that boundary. The bundle contract and handoff docs therefore need to own portability and cache identity instead of pushing those concerns to consumers.
- **Existing patterns to reuse**: `doc_web/preview.py`, `doc_web/preview_pdf.py`, `doc_web/preview_bundle.py`, `doc_web/preview_support.py`, `DocWebBundleManifest`, `DocWebPreviewMetadata`, `DocWebPreviewSelectorMap`, `tests/test_doc_web_preview.py`, and the existing Story 228 preview schema-validation pattern.
- **Eval**: Focused pytest proves the fixture, fallback, hard-failure path, safe-file contract, and cache identity. Schema validation and manual inspection of `output/runs/story229-preview-contract-validation/output/html/` prove the emitted artifacts.

## Architectural Fit

- **Owning module / area**: `doc_web` preview runtime and preview bundle contract. PDF page fallback belongs in `doc_web/preview_pdf.py`; portable replay/cache identity belongs in `doc_web/preview_bundle.py`, `doc_web/preview_identity.py`, `doc_web/preview_support.py`, and the Pydantic schemas.
- **Methodology reality**: This serves the intake, extraction/OCR, validation/provenance, and Dossier handoff categories (`spec:1`, `spec:3`, `spec:6`, `spec:7`). It does not change final maintained format coverage because preview mode remains a non-final runtime surface.
- **Substrate evidence**: Story 228 already provided `doc-web preview`, preview schemas, selector map, cache identity, and sample validation. Current-pass substrate verified by code/tests: `doc_web/preview_pdf.py`, `doc_web/preview_bundle.py`, `doc_web/preview_identity.py`, `schemas.py`, `tests/test_doc_web_preview.py`, and the generated mixed fixture.
- **Data contracts / schemas**: `DocWebBundleManifest` now includes `files: list[DocWebBundleFile]`. `DocWebPreviewMetadata` embeds `DocWebPreviewCacheIdentity`, `validate_artifact.py` validates `doc_web_cache_identity_v1`, and `doc-web contract --json` exposes `cache_identity` in supported preview schema versions and preview contract fingerprinting.
- **File sizes**: `make check-size` passes. Extracting identity assembly to `doc_web/preview_identity.py` keeps `doc_web/preview.py`, `doc_web/preview_pdf.py`, and `doc_web/preview_bundle.py` below the repo's 400-line size threshold.
- **Decision context**: ADR-002, `docs/doc-web-bundle-contract.md`, and `docs/dossier-doc-web-handoff.md` were reviewed. ADR-003 remains background only; this work strengthens the accepted ADR-002 bundle boundary rather than reopening the Docling boundary decision.
