## Goal

Process `/Users/cam/Documents/Projects/doc-web/input/f5-robo-rally-rulebook.pdf` into a strongly semantic, plain HTML manual that preserves rule text, reading order, and essential explanatory graphics with source provenance. The story should use this rulebook as a hard-case pilot for a reusable class of print-ready, graphic-designed manuals: imposed landscape spreads, dense visual layout, explanatory diagrams, board/course maps, icon/card reference material, and a first PDF page that contains the back page plus the front page. Success is not a one-off Robo Rally scraper. Success is a generic detection and validation path that can recognize this document shape, infer logical page order, decide which graphics are essential to the rules, and produce inspectable plain HTML artifacts without hiding policy in prompts.

## Eval Ladder Context

- **Root eval / ideal path**: no current root eval proves arbitrary graphics-heavy manuals can become Dossier-ready semantic HTML with essential figures, logical page order, and block-level provenance.
- **Parent evidence**: `born-digital-pdf` is currently passing only on bounded book-like and flat fixtures, and its coverage row explicitly says image-heavy or unproven born-digital PDFs may still return `no-recipe-needed`. Story 009 proved semantic figure placement on Onward but deferred multi-column/complex layout until real failures appeared. Story 207 refreshed crop detection proof to `0.9703`, but that surface measures illustration crop quality, not full manual semantic reconstruction or print imposition.
- **Measured hard-case evidence in this pass**: `pdfinfo` reports an untagged PDF/X-4, 16 physical pages, landscape `1302 x 870 pts`; `pdftotext -layout` exposes physical spread labels `32-1`, `2-3`, `4-5`, ..., `30-31`; `pdfimages -list` reports 1180 image entries across the 16 physical pages, including 108 large entries; `extract_pdf_images_fast_v1` produced a valid 16-row `page_image_v1` manifest; `split_pages_from_manifest_v1` split it into 32 valid page-image rows with spread confidence `0.96`, but the emitted order is physical split order (`32, 1, 2, 3, ...`) rather than logical manual order.
- **Child eval for this story**: a bounded rulebook semantic-hardcase evaluation should score logical page mapping, reading order, rule-text fidelity, essential-graphics coverage, figure captions/alt text, and provenance on representative pages from this PDF. The first implementation pass should decide whether that child eval can be a story-local artifact, a maintained promptfoo task, or a reusable driver-backed conformance report.

## Approach Evaluation

- **Simplification baseline**: test whether a strong multimodal model can already produce acceptable semantic plain HTML plus essential-graphic decisions from a representative spread or a small logical-page slice. If a single-call or per-spread baseline works, the implementation should shrink to orchestration, provenance, and validation artifacts. If it fails, the failure mode should determine whether the next step is page-order inference, crop/graphics classification, semantic reconstruction, or conformance scoring.
- **AI-only**: plausible for semantic understanding and visual-role classification, but risky as the full solution because the PDF is untagged, imposed, image-dense, and needs exact provenance. An AI-only pass must still emit page-map, graphic-inventory, and conformance artifacts, not just final HTML.
- **Hybrid**: likely strongest but still must be proven. Use deterministic PDF render/split and page-label/order checks for the mechanical substrate, VLM/LLM calls for semantic HTML and graphic role classification, and deterministic validators for page coverage, duplicate/missing logic, provenance, and artifact completeness.
- **SOTA visual planner**: for short, heavily graphic-designed manuals, a flagship multimodal model should inspect each logical page image and produce a manifest of essential visuals before cropping. This should improve semantic intent over OCR placeholder counts while remaining generic: the model decides what matters and why, code performs source-pixel crops, provenance, and validation. AI-generated grid/contact-sheet images are useful as diagnostic artifacts, but they are not acceptable final figure assets because they can introduce hallucinated pixels.
- **Pure code**: useful for rendering, split diagnostics, page-map consistency checks, file packaging, and conformance reports. It is not enough for deciding which explanatory graphics are semantically essential or for repairing complex reading order in a heavily designed manual.
- **Repo constraints / prior decisions**: ADR-002 keeps the output boundary at semantic HTML plus provenance. The consistency runbook says policy must be explicit via artifacts such as inventories, plans, and reports. Crop and layout stories prove reusable components exist, but they do not cover imposed rulebooks end to end. Cost discipline requires OCR artifact reuse during iteration.
- **Existing patterns to reuse**: `extract_pdf_images_fast_v1`, `split_pages_from_manifest_v1`, `modules/common/image_utils.py`, `ocr_ai_gpt51_v1` figure/figcaption prompt and sanitizer, `crop_illustrations_guided_v1`, `crop_illustrations_from_ocr_v1`, `build_chapter_html_v1`, `doc_web_bundle_manifest_v1`, `doc_web_provenance_block_v1`, the crop eval runbook, Story 009's semantic figure placement pattern, and Story 154's Dossier handoff contract.
- **Eval**: the deciding eval should compare at least three candidates on the same representative slice: existing PDF/OCR pipeline baseline, strong VLM direct semantic baseline, and a hybrid deimpose + semantic/classification pipeline. Passing means the candidate preserves logical order, rule text, essential graphics, and provenance with inspectable uncertainty.
